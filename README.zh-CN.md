# Karpathy 风格 Claude Code 准则

> 看看我的新项目 [Multica](https://github.com/multica-ai/multica) — 一个用于运行和管理编码 Agent 及可复用技能的开源平台。
>
> 关注我的 X：[https://x.com/jiayuan_jy](https://x.com/jiayuan_jy)

一个 `CLAUDE.md` 文件，用于改善 Claude Code 的行为，灵感来源于 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876)——关于 LLM 编程中的常见陷阱。

## 问题所在

来自 Andrej 的观察：

> "模型会替你做出错误的假设，然后一路狂奔而不去验证。它们不会管理自己的困惑，不会寻求澄清，不会发现不一致之处，不会展示权衡方案，该反对的时候也不会反对。"

> "它们特别喜欢把代码和 API 搞得过于复杂，抽象膨胀，不清理死代码……本来 100 行就能搞定的事，非要写出 1000 行的臃肿实现。"

> "它们有时仍会作为副作用去修改/删除自己不够理解的注释和代码，即使这些改动与当前任务无关。"

## 解决方案

一个文件中的四条原则，直击这些问题：

| 原则 | 解决的问题 |
|------|-----------|
| **编码前先思考** | 错误假设、隐藏的困惑、遗漏的权衡 |
| **简洁优先** | 过度复杂化、臃肿的抽象 |
| **精准修改** | 无关的改动、不该碰的代码被修改 |
| **目标驱动执行** | 通过测试先行和可验证的成功标准来获得杠杆效应 |

## 四大原则详解

### 1. 编码前先思考

**不要假设。不要隐藏困惑。暴露权衡。**

LLM 经常悄悄选择一种理解然后就照着做了。这条原则强制要求显式推理：

- **明确表达假设** — 不确定就问，不要猜
- **展示多种解读** — 有歧义时不要默默选一个
- **该反对时就反对** — 如果有更简单的方案，就说出来
- **困惑时就停下** — 说清楚哪里不明白，然后提问

### 2. 简洁优先

**解决问题的最少代码。不写投机性代码。**

对抗过度工程化的倾向：

- 不做超出需求的功能
- 一次性代码不做抽象
- 没有要求就不加"灵活性"或"可配置性"
- 不为不可能的场景做错误处理
- 如果 200 行可以缩到 50 行，就重写

**检验标准：** 一个资深工程师会说这段代码过于复杂吗？如果是，就简化。

### 3. 精准修改

**只动该动的。只清理自己造成的垃圾。**

编辑已有代码时：

- 不要"顺手改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你会用不同的方式写
- 发现无关的死代码，提一下就好——不要删

当你的改动产生了孤立代码时：

- 移除因为你的改动而变得无用的 import/变量/函数
- 不要删除改动之前就已存在的死代码，除非被要求

**检验标准：** 每一行变更都应该能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将指令性任务转化为可验证的目标：

| 原始说法... | 转化为... |
|------------|----------|
| "添加验证" | "为无效输入编写测试，然后让测试通过" |
| "修复这个 bug" | "编写复现 bug 的测试，然后让测试通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，写一个简短的计划：

```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强有力的成功标准让 LLM 可以自主循环。模糊的标准（"让它能跑"）需要不断澄清。

## 安装

**方式 A：Claude Code 插件（推荐）**

在 Claude Code 中，先添加 marketplace：
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

然后安装插件：
```
/plugin install andrej-karpathy-skills@karpathy-skills
```

这将把准则作为 Claude Code 插件安装，使技能在所有项目中可用。

**方式 B：CLAUDE.md（单项目）**

新项目：
```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

已有项目（追加）：
```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

## 核心洞察

来自 Andrej：

> "LLM 非常擅长循环直到达到特定目标……不要告诉它该做什么，给它成功标准，然后看它自己搞定。"

"目标驱动执行"原则正是捕捉到了这一点：将命令式指令转化为声明式目标，配合验证循环。

## 如何判断准则是否在起效

如果你看到以下情况，说明准则在起作用：

- **diff 中不必要的改动更少了** — 只出现被请求的变更
- **因过度复杂而返工更少了** — 代码第一次就很简洁
- **澄清问题出现在实现之前** — 而不是在犯错之后
- **PR 干净简洁** — 没有随手重构或"改进"

## 自定义

这些准则设计用于与项目特定指令合并。添加到你已有的 `CLAUDE.md` 中，或创建一个新的。

对于项目特定规则，添加类似这样的章节：

```markdown
## 项目特定准则

- 使用 TypeScript 严格模式
- 所有 API 端点必须有测试
- 遵循 `src/utils/errors.ts` 中的现有错误处理模式
```

## 权衡说明

这些准则偏向于**谨慎而非速度**。对于简单任务（修正拼写错误、显而易见的一行改动），请自行判断——不是每次改动都需要全套流程。

目标是减少非简单工作中的高代价错误，而不是拖慢简单任务。
